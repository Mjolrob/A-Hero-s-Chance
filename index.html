The Glossary

HTML (HyperText Markup Language)
The standard markup language for websites. Usually paired with CSS and JavaScript for more advanced web pages.
CSS (Cascading Style Sheets)
The style sheet language that is used for the presentation of a document. This is written in a markup language like HTML.
JavaScript
This is a script language that makes web page more dynamic and interactive.
Elements
These consist of a start tag and an end tag, with the content inserted within the tags.
Tags
These are elements that are inserted into a HTML document which adds and defines the content of a web page.
Attributes
These provide additional information about HTML elements and are always specified in the start tag.

Week 6: Becoming Responsive

Edited on November 5th, 2019

To make the website responsive, certain changes needed to happen and certain additions needed to be added, so the sixth week was about:

  1. Adding the viewport meta element to the head of the document
  2. Organising the style sheet
  3. Setting up the responsive grid
  4. Styling the small screen layout (mobile first approach)
  5. Styling for different screen widths
  6. Hiding elements (display:none)
  7. Making images flexible
  8. Testing the responsiveness of the web pages

The viewport meta tag

The sixth week meant it was time to take the website’s responsiveness to the next level from how it was before. Since my web pages already had meta name="viewport" content="width=device-width, initial-scale=1" which allows devices to render the website at their own screen widths and not that of an average desktop. I also used my phone, some other peoples (friends and family) devices and also MobileTest to check my website was good at different devices and screen sizes (which I will check regularly). Chrome and Firefox also have built-in developer tools than you can use to resize your page to simulate different devices, which I will also use going forth.

Responsive grid

Now it was time to set up the responsive grid. The responsive grid makes it makes it easier to place elements on the page and allows sections of the web pages to be placed into segments which can slide into different positions depending on the screen size (which provides an excellent, unique viewing experience on tablets and mobile devices). Jennie provided the HTML and CSS that was needed to make websites responsive which I utilised well. However, I had to spend a large chunk of time just rearranging and merging my old CSS stylesheet to the new one provided as they both had different layouts and the new one even had new features added in! Although it was a bit of trouble to add in, the new layout is definitely a lot better to have as it seems less like a page from pre-2010!

I also went with the ‘mobile first approach’ which has the mobile devices in mind first as mobile devices tend to have more restrictions and is easier to expand the website onto tablets or desktops rather than regress down to mobile.

Hiding the cursed table

On small mobile screens, the table is hidden due to its clumsiness and size working poorly on smaller devices.

Make images flexible

The next leap to make the website responsive was to make the images flexible, which means that the images resize themselves based on the size of their parent element (which is the code max-width: 100%;) and won’t stretch off the page jarringly on smaller devices due to it resizing with its original aspect ratio. Due to this being done with CSS, the HTML height and width properties were no longer needed and were taken out.

Week 5: Even more CSS?

Edited on October 29th, 2019

The CSS is added in but now in the fifth week it needs a boost, as well as me focusing on other pages. So, I have:

  1. Begun to create the Tutorial page, which includes:
    • Adding meta name="viewport" content="width=device-width, initial-scale=1"

  2. Personalising the CSS by:
    • Specifying the font
    • Changing line-heights, margins, text-indents, text-aligns and more

Expanding the CSS

When I added the CSS in the previous week, all I did was change the sizes of some elements and changes a few colours on the background and such but now it was time to get really specific and make the small changed that add up. This includes me:

  • Changing the main font, with secondary fonts as a safety precaution
  • Styling the text (margins, indents, aligning text, heights et cetera)
  • Floating certain images

The font

The font that I chose to go for is Segoe UI which is used by Microsoft. The reason for this is because, which I found out during my independent research, it is a comforting Sans-serif font with the text being soft and it was designed to be optimised for user interfaces (with its wide vertical gaps) and readable text strings across multiple languages. Microsoft themselves describe Segoe UI as

‘an approachable, open, and friendly typeface, and as a result has better readability than Tahoma, Microsoft Sans Serif, and Arial.’
Microsoft also said that it was
‘designed to be a humanist sans serif with no strong character or distracting quirkiness.’

Styling the text

With the line-height, I made it fairly spacious without it look like the sentences were trying to run away from each other since there is nothing worse than text spacing seeming off or improper as it can — and will — distract the viewer and put them off. I created space between the blocks of text with the margin property margin: 10px 10px 10px 10px; so everything isn’t so cramped together and unnecessarily tight. Jennie explained in Tutorial 5 that, if there are several paragraphs of text on a page, then text-indents can improve legibility as it indents the first line of each paragraph. I tested this out and decided that it wasn’t for my website as I have a lot of new lines and it makes everything very disorientating (and I break up the text with images and strong p heading as well to improve legibility).

Image positioning

Floating the images is another tool in the box to make the website visually pop out and have a more professional vibe as floating an image allows the text to wrap around it, which make the page seem more streamlined and less ‘blocky’. To do this, I had to create new classes in my CSS file which were .fltlt (float left), .fltrt (float right) and .clear (clears the float at a specified point so it doesn’t keep on going and unwantedly affect the web page) (the names were provided by Jennie but they are relevant and easy to remember so I didn’t change them) and then place the img into a div with the code div class="fltlt/flrrt" and clearing the float with class="clear".

Some images flow better without floating around text, so while I won’t use the float sparingly, I will make decision on where and not to use it.

I mainly looked at Microsoft’s own sans-serif font ‘Segoe’ as I’ve always owned computers that use Microsoft Windows OS and I haven’t yet come across a clearer, easier and better-looking font. I more specifically centred in on ‘Segoe UI’ that Microsoft use for their user interfaces and looked on Microsoft’s own website to find out more about the font and to find out why they created it and what makes it special (which I quoted in ‘The Font’ section of Week 5). Since they created it to be accessible, friendly and have superior readability I thought that it would be a no-brainer to use it as my primary font.

Week 4: The Style

Edited on October 22nd, 2019

When we hit the fourth week, it was time to add the missing piece to my website. It was time to really give it the nutrients it needed: CSS. I did this by:

  1. Linking normalise.css to my index
  2. Linking stylesheet.css to my index
  3. Styling the body (and other elements) of my web pages

Normalising the pages

Before adding stylesheet.css to my web pages, I needed to add something that is equally as important in improving my website: normalise.css. This is a stylesheet that gives cross-browser consistency with rendering the elements of a web page and does it within modern standards. To get normalise.css (version 8.0.0) I went to the link provided by Jennie, and to add it to the pages all I had to do was add link href="css/normalize.css" rel="stylesheet" into the head section of my web pages.

The CSS

Afterwards it was time for the main event, the stylesheet.css. Jennie provided a basic template for me to use where the tags were arranged into a logical order for easy editing.

Editing the stylesheet was going to be very trial and error sue to me trying to figure out what is what and how changing X will affect Y and so forth. I made the body colour a nice light grey (#f7f7f7) and the wrapper colour white (#ffffff). The reason I did this was because any colour can go with white and light grey so if I wanted to add red, blue, purple or anything then I don’t have to worry about matching colours. I made the font size use em instead of px due to it complimenting responsive websites much better thanks to Jennie’s input when I asked her about the difference between font-size types.

I made the heading all the same shade of colour; which is blue (could change in the future). This is so that there isn’t too many colours clashing with each other and with a (fairly) simple website that I am making, it doesn’t need a large array of colours overwhelming the page.

Since the stylesheet.css is ever-changing, it will be difficult to write about everything I am changing since it will be constant tweaks and additions, but any large noteworthy changes will be documented in my Learning Journal in the weeks ahead.

Validating the CSS

You can validate the CSS like you can with the HTML from here to which I will do frequently to make sure I am being efficient and proper with my increasing website development skills.

Independent Research

Since this week was all to do with the CSS, I wanted to put in a lot of time to help me really grasp what CSS can do and how I can use it efficiently. I began with looking at the CSS basics from Mozilla and absorbing the information such as what declarations and property values are. It goes through the fundamentals in a comprehensive manner which is exactly what I wanted (such as fonts and box models).

I then started reading through font styling and how you can manipulate text with colour, size (px, em and rem), types of fonts (serif, sans-serif, monospace, cursive and fantasy), common fonts (Arial, Courier New, Times New Roman, Verdana) and so on. I decided to use em after talking to Jennie and through my independent research as although rem is an ‘easier’ em, it is not supported by Internet Explorer 8 and below which is an issue as my website is supposed to work on all devices. I also learnt of font-style which is the italic text and the oblique text (which I never heard of before), font-weight which is bold and bolder, and font-decoration which is underlining text, overlining text or shoving a line though the text.

I stumbled upon a way that you can give text multiple shadows, although this looked hideous and cheap to me so I was immediately put off from it.

The website has a grand list of random text layout styles such as:

  • text-overflow which defines how overflowed content that can't be shown is signalled to viewers
  • word-break which defines whether to break lines within words
  • direction which tells the text direction (although this depends on the language)
  • hyphens which toggles hyphenation (for supported languages)
  • text-orientation: which defines the orientation the text
  • overflow-wrap which tells the browser whether or not it can break lines within words in order to prevent overflow off the web page

I also looked up a tiny bit about class selectors due to me wanting to arrange my images in a certain place (which is called floating) but as that will be covered next week, I didn’t worry too much about going crazy with it right now. When I got stuck on how to use and edit CSS segments, I would use Mozilla's CSS reference to help me understand how whatever I was trying to do worked and could be manipulated so that I could remember it for the future.

Week 3: The Cursed Tables

Edited on October 15th, 2019

The third week came and a third go at growing my website came with it. Week 2 was mainly about lists and tags, and Week 3 would carry this on with some extra additions to go along with it.

Week 3 concluded and the index, once again, improved drastically with new features. These new features were:

  1. Adding two new types of lists: nested lists (a combination of lists/a list inside a list) and description lists (what it says on the tin)
  2. Adding a table — or more specifically my University time table — at the top of the home page
  3. Adding quote text
  4. Transforming an image into links to external websites

Even more lists!

The third week started with most lists: the nested list and the definition list (that I looked up before in Week 2). Nested lists are just where you put a list into a list so that was fine for me as I had already done that in my test environment in Week 2.

I hadn’t done description lists before, although I did come across them during my previous independent research. They are a bit more complex sine you need three tags instead of just two (with ordered and unordered lists) which are dl (starts the description list), dt (the terms), dd (the description). Going forth, I will only use these for their named purpose for descriptions of jargon and noteworthy content.

The pain known as tables....

Before we started the table, Jennie told us that tables were annoying, fiddly and just generally difficult to make and debug. I thought that it couldn’t have been that bad, but if anything, she undersold it….

Due to what Jennie said, I immediately threw together a basic table in my test environment to get a feel of how they work. After a short amount of time I could see how much if a pain they will be when trying to add columns and rows and then fill them in with meaningful data. It’s bad enough that there are four tags to use but when you need to write downwards for the data to go into the neighbouring row, it can get a bit messy. I made a few errors with where my data was placed in the tables (data that I wanted to be placed in a column would be placed in a row and vice versa) but after taking it slow and evaluating how tables work I eventually became proficient in getting it right.

Jennie said in Tutorial 3 that ‘it's a good thing that we no longer need to use tables for layout’ and she is definitely right!

Quoting the sources

The next step was to include the sources I use that helped me along the journey or my Learning Journal but not as hyperlinks but as quotes. The quotes can come from many sources such as from a website, a book, a magazine or even my lecturer Jennie!

Due to me potentially using many different sources during my work, it is vital that I keep up to date with this and don't ignore it!

Transforming the images

The final addition of the third week was making an image (or images in my case with the glossary) into a link. This was rather simple to do as all you needed to do was add in a /a and href.

Independent Research

There wasn’t much independent learning for me as I grasped the third week very well and didn’t struggle at all. Although with that in mind, I did have to look up in depth how to do tables as they were just a pain when I started to do my timetable. To do this I used two resources from W3Schools, which were the web pages 'W3Schools HTML tables' and 'W3Schools HTML table tag' as they both went in depth with examples on how tables work.
